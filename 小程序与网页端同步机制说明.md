# 小程序与网页端同步机制说明

本文档用于说明货源看板小程序与网页端之间的数据同步机制、接口约定、字段映射、当前已知问题和排查建议。

## 1. 同步接口

小程序端统一通过 `services/cloud-api.js` 调用网页端接口。

接口地址：

```text
https://21068849.xyz/api/mini/items
```

请求头：

```http
content-type: application/json
x-juzhen-token: <JUZHEN_MINI_PROGRAM_TOKEN>
```

相关接口：

```text
GET    /api/mini/items?side=supply&includeDeleted=1
GET    /api/mini/items?side=demand&includeDeleted=1
POST   /api/mini/items
PUT    /api/mini/items/:id
DELETE /api/mini/items/:id
```

小程序端期望 `GET` 返回：

```json
{
  "items": []
}
```

小程序端期望 `POST/PUT` 返回：

```json
{
  "item": {}
}
```

如果 `POST/PUT` 没有返回 `item`，小程序会使用本地提交前的数据作为同步成功后的本地对象。

## 2. 小程序拉取网页端数据

看板页 `pages/board/index.js` 在 `onShow` 时会触发同步：

```js
Promise.all([service.refreshCloudItems(), demandService.refreshCloudDemands()])
```

货源拉取链路：

```text
pages/board/index.js
-> services/inventory.js refreshCloudItems()
-> services/cloud-api.js fetchItems("supply")
-> GET /api/mini/items?side=supply&includeDeleted=1
-> mapRemoteSupply()
-> mergeRemoteItems()
-> saveItems()
-> wx.setStorageSync("inventory_board_items", ...)
```

需求拉取链路：

```text
pages/board/index.js
-> services/demand.js refreshCloudDemands()
-> services/cloud-api.js fetchItems("demand")
-> GET /api/mini/items?side=demand&includeDeleted=1
-> mapRemoteDemand()
-> mergeRemoteDemands()
-> saveDemands()
-> wx.setStorageSync("inventory_board_demands", ...)
```

当前失败现象：

```text
setStorageSync fail
```

这通常说明接口可能已经返回数据，但小程序写本地缓存失败，导致页面没有更新。

## 3. 小程序提交数据到网页端

新增/编辑货源提交链路：

```text
pages/form/index.js submitForm()
-> services/inventory.js upsertItemToCloud()
-> services/inventory.js upsertItem()
-> services/cloud-api.js saveSupply()
-> uploadMediaFields()
-> toRemoteSupply()
-> POST /api/mini/items 或 PUT /api/mini/items/:id
```

新增/编辑需求提交链路：

```text
pages/demand-form/index.js
-> services/demand.js upsertDemandToCloud()
-> services/cloud-api.js saveDemand()
-> toRemoteDemand()
-> POST /api/mini/items 或 PUT /api/mini/items/:id
```

## 4. 货源字段映射

### 4.1 小程序提交给网页端的货源字段

由 `services/cloud-api.js -> toRemoteSupply()` 生成。

```js
{
  id,
  side: "supply",
  category,
  condition,
  title,
  quantity,
  pricing,
  person,
  phone,
  customer,
  ownerInfo,
  urgent,
  note,
  image,
  video,
  scope,
  ownerName,
  machineConfig
}
```

关键字段说明：

```text
side: 固定为 "supply"
title: 货源标题
quantity: 字符串形式数量
pricing.saleEnabled: 是否出售
pricing.salePrice: 出售价格
pricing.saleUnit: "万元/台"
pricing.rentalEnabled: 是否租赁
pricing.rentalQuotes: 租赁报价
person: 当前小程序登录人姓名
phone: 当前小程序登录人手机号
ownerName: 当前小程序登录人姓名
ownerInfo: 货主信息，当前已经作为顶层字段发送
note: 备注，同时会拼接扩展字段
scope: "company" / "mine" / "shared"
image: 第一张图片 URL
video: 视频 URL
machineConfig: 机器配置
```

货源类型映射：

```text
小程序 sourceType = "company" -> 网页端 scope = "company"
小程序 sourceType = "own"     -> 网页端 scope = "mine"
小程序 sourceType = "shared"  -> 网页端 scope = "shared"
```

货主信息兼容写法：

```text
当前小程序会发送顶层 ownerInfo。
同时 note 中也会包含一行：货主信息：xxx
```

网页端建议优先保存和返回顶层 `ownerInfo`。

### 4.2 网页端返回给小程序的货源字段

由 `services/cloud-api.js -> mapRemoteSupply()` 解析。

小程序当前读取：

```js
{
  id,
  title,
  summaryTitle,
  category,
  condition,
  quantity,
  price,
  pricing,
  person,
  phone,
  customer,
  ownerInfo,
  note,
  scope,
  image,
  video,
  machineConfig,
  createdAt,
  updatedAt,
  deleted,
  deletedAt
}
```

关键解析逻辑：

```text
title = item.title || item.summaryTitle || "未命名货源"
price = item.pricing.salePrice || item.price || ""
ownerInfo = item.ownerInfo || 从 note 中解析 "货主信息："
sourceType = scope === "company" ? "company" : scope === "shared" ? "shared" : "own"
imageUrls = 只有 item.image 是 http/https URL 时才缓存
mediaFiles = 只有 image/video 是 http/https URL 时才缓存
```

网页端返回图片/视频时必须注意：

```text
不要在列表接口中返回 base64。
image/video 字段建议只返回 http/https URL。
如果返回 base64，小程序端现在会丢弃该媒体字段，避免 setStorageSync 超限。
```

## 5. 需求字段映射

### 5.1 小程序提交给网页端的需求字段

由 `services/cloud-api.js -> toRemoteDemand()` 生成。

```js
{
  id,
  side: "demand",
  category,
  condition,
  title,
  quantity,
  pricing,
  person,
  phone,
  customer,
  urgent,
  note,
  image,
  video,
  scope,
  ownerName
}
```

需求目前没有 `ownerInfo`。

### 5.2 网页端返回给小程序的需求字段

由 `services/cloud-api.js -> mapRemoteDemand()` 解析。

小程序读取字段包括：

```text
id, title, summaryTitle, category, quantity, pricing, person, phone,
customer, note, deliveryDate, scope, image, video, createdAt, updatedAt,
deleted, deletedAt
```

需求扩展字段从 `note` 中解析：

```text
期望交期天数
地区
使用场景
可接受成色
发票要求
付款方式
决策期限
```

## 6. 本地缓存机制

小程序目前不是纯远端渲染，而是拉取远端数据后写入本地缓存，再从缓存生成看板列表。

货源缓存 key：

```text
inventory_board_items
```

需求缓存 key：

```text
inventory_board_demands
```

日志缓存 key：

```text
inventory_board_logs
inventory_board_demand_logs
```

目前已做的缓存瘦身：

```text
1. 只缓存 http/https 图片 URL。
2. 丢弃 base64、本地临时路径和非 URL 媒体字段。
3. setStorageSync 失败时先 removeStorageSync 再写入。
4. 仍失败时降级保存最小字段，并清空 imageUrls/mediaFiles。
```

如果仍然出现：

```text
setStorageSync fail
```

说明可能仍有以下问题：

```text
1. 远端列表一次返回数据量过大。
2. 单条数据字段过大，例如 note、machineConfig、pricing、其他未预期字段很大。
3. 小程序本地其他 key 占用了总缓存空间。
4. 微信小程序本地缓存总限制无法通过代码提升。
```

## 7. 当前已知风险点

### 7.1 网页端新增数据不同步到小程序

排查重点：

```text
1. GET /api/mini/items?side=supply&includeDeleted=1 是否返回了网页端新增的记录。
2. 返回 JSON 顶层是否是 { items: [...] }。
3. 新增记录是否有 id。
4. 新增记录 side/scope 字段是否符合约定。
5. image/video 是否为 URL，不能是 base64。
6. 小程序是否仍报 setStorageSync fail。
```

### 7.2 小程序新增/编辑不同步到网页端

排查重点：

```text
1. POST/PUT /api/mini/items 是否收到请求。
2. x-juzhen-token 是否校验通过。
3. 请求 body 中 side 是否为 "supply" 或 "demand"。
4. 请求 body 中 ownerInfo 是否被网页端保存。
5. POST/PUT 响应是否返回 { item: ... }。
6. 网页端保存后 GET 列表是否能返回该记录。
```

### 7.3 货主信息不同步

小程序当前同时兼容两种格式：

```json
{
  "ownerInfo": "货主信息内容",
  "note": "备注\n货主信息：货主信息内容"
}
```

网页端建议：

```text
1. 保存顶层 ownerInfo。
2. GET 返回顶层 ownerInfo。
3. 如果历史数据只有 note 中的 "货主信息："，网页端可自行兼容解析。
```

### 7.4 货源类型不同步

小程序枚举：

```text
company = 公司货源
own     = 我的货源
shared  = 共享给我
```

接口映射：

```text
company <-> scope: "company"
own     <-> scope: "mine"
shared  <-> scope: "shared"
```

网页端需要支持 `scope = "shared"`，否则“共享给我”会丢失或被当成普通自有货源。

## 9. 建议排查步骤

1. 在网页端新增一条不带图片/视频的最小货源。
2. 用浏览器或 Postman 请求：

```text
GET https://21068849.xyz/api/mini/items?side=supply&includeDeleted=1
```

确认返回中有该记录。

3. 确认返回结构是：

```json
{ "items": [...] }
```

4. 确认单条记录包含：

```text
id, title, side/scope, quantity, pricing 或 price, person, phone, createdAt, updatedAt
```

5. 确认 image/video 不包含 base64。
6. 小程序端进入看板页，观察是否仍提示 `setStorageSync fail`。
7. 如果仍失败，建议临时让网页端接口只返回最新 20 条货源，验证是否为数据量问题。
8. 如果最新 20 条可同步，再做分页或小程序端改为远端数据直接渲染，不缓存整包列表。

## 10. 建议后续改造

当前机制仍依赖本地 storage 保存完整看板数据，数据量增长后不稳定。

建议改造为：

```text
1. 看板页远端分页拉取，直接 setData 渲染。
2. storage 只保存当前用户、筛选条件、少量最近访问记录。
3. 列表接口分页，例如 page/pageSize 或 cursor。
4. 图片/视频走独立 URL，不进入列表 JSON 大字段。
5. POST/PUT/GET 使用同一套字段 schema，避免 note 扩展字段长期承担结构化数据。
```
