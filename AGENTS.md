## 核心工作原则

### 1. 先规划，后执行

收到任何非平凡任务后，按以下顺序推进：

1. 商业分析：从第一性原理出发，思考产品战略、差异化、增长路径、风险与反脆弱策略。
2. 理解需求：确认任务目标、边界条件、验收标准。有歧义时先澄清。
3. 方案设计：设计前先查阅项目现有代码、架构和约定，复用已有实现；确需新建时说明理由。
4. 任务拆分：拆成可独立验证的小模块，每个模块不超过 3-5 个文件变更。
5. 逐模块交付：完成一个模块、验证通过、再进入下一个。

复杂任务主动建议使用 `/spec` 或 `/plan` 工作流。

### 2. 决策置信度管理

每个技术决策点都要判断置信度：

- 置信度 >= 90%：直接执行，在代码注释或说明中简述关键理由。
- 置信度 80%-90%：可以执行，但要向用户说明不确定性和判断依据。
- 置信度 < 80%：暂停，先通过项目代码、官方文档或权威资料查证；仍不足时向用户确认。

禁止在不确定时猜测接口签名、框架 API、业务字段或安全配置。

### 3. 事实查验

以下信息必须查验，不得凭记忆臆测：

- 微信小程序 API、云函数、云开发行为和端差异。
- 第三方库或框架 API、版本兼容性、breaking changes。
- 业务规则、数据字段含义、同步接口约定。
- 安全相关配置、Token、短信和云服务设置。

查验优先级：

1. 项目现有代码。
2. 项目文档和 `rules/` 规则。
3. 官方文档。
4. 权威技术文章。
5. 社区讨论。

### 4. 代码质量标准

- 每次改动后运行相关测试或做最小验证。
- 函数保持单一职责，超过 30 行时考虑拆分。
- 注释解释“为什么”，不要重复描述“做什么”。
- 避免过度抽象：首次出现不急于提取，第二次考虑提取，第三次必须提取。
- 错误处理必须显式，不允许吞掉异常或返回空值而不记录。
- 优先使用项目内已有工具、服务和函数，减少重复实现。

### 5. 成本与效率

- 查找文件优先用 `rg` / Glob，搜索内容优先用 Grep，不通读整个项目。
- 读取文件按任务需要读取相关范围，不为了“全面了解”而扩大上下文。
- 独立文件读取和搜索可以并行，但并行数量不超过 5 个。
- 输出简洁，直接给结论和关键差异，不展示完整文件，除非用户明确要求。
- 不主动添加用户未要求的额外功能或“以后可能用到”的扩展。

## 架构与模块规则

- 做架构决策前先查项目结构、相似实现、README 和规则文件。
- 只有确认现有代码无法满足需求时，才新建模块或引入依赖。

- 依赖方向保持单向：工具函数不依赖业务模块，业务逻辑不直接依赖 UI 内部实现，公共组件不直接引用页面路径。

- 禁止循环依赖和直接操作其他模块内部状态。

引入新依赖前必须检查必要性、成熟度、包体积、小程序兼容性和许可证。

## 工作流与验收规则

任务复杂度判断：

| 复杂度 | 特征 | 策略 |
| --- | --- | --- |
| 低 | 单文件修改、修 bug、加注释 | 直接执行，完成后自检 |
| 中 | 2-5 个文件、一个功能模块 | 使用 `/plan` 思路拆分 2-3 个子任务 |
| 高 | 新模块、跨层变更、架构调整 | 使用 `/spec` 思路生成完整规范 |

每个子任务应：

- 涉及文件不超过 5 个。
- 可在一个步骤内完成编码和验证。
- 有明确、可测试的完成标准。

每个子任务完成后做：

1. 代码审查：命名、逻辑、边界情况。
2. 编译检查：确认无语法或类型错误。
3. 功能验证：运行相关页面、单元测试或手动路径。
4. 回归检查：至少做一次 smoke test。

任务过大时，主动建议拆分为多个对话或先交付 MVP。

## 质量闸门

禁止编造以下内容：

- 不存在的 API、函数、类、方法名。
- 不存在的 npm 包、库、插件。
- 不存在的配置项、参数、选项。
- 不存在的文件路径或目录结构。
- 不存在的错误码、错误信息。

代码规范：

- 变量和函数使用有意义的英文单词，避免拼音和非通用缩写。
- 布尔变量以 `is`、`has`、`should`、`can` 开头。
- 事件处理函数以 `handle` 或 `on` 开头。
- 常量使用全大写蛇形命名。
- 函数参数不超过 4 个，超过时考虑配置对象。
- 避免布尔参数作为行为开关。
- 所有异步操作必须有错误处理。
- 所有用户输入必须校验。
- 不允许空 `catch`；确实忽略时必须说明原因。
- 禁止注释掉的代码块、生产遗留 `console.log` 和魔法数字。

## 小程序与网页端同步规则

涉及货源/需求字段、新增字段或接口变更时，必须同时检查：

1. 网页端 `cleanItem()` 是否保存字段。
2. 网页端 `publicItem()` / `miniItem()` 是否返回字段。
3. 小程序 `toRemoteSupply()` / `toRemoteDemand()` 是否提交字段。
4. 小程序 `mapRemoteSupply()` / `mapRemoteDemand()` 是否解析字段。

新增货源/需求字段还要检查：

- `services\cloud-api.js`
- `services\inventory.js` 或 `services\demand.js`
- `pages\form` 或 `pages\demand-form`
- 必要时检查 `pages\detail`、`pages\board`、`pages\batch-import`
- 导入/导出表头、老数据迁移默认值、卡片展示、详情展示、权限过滤后的返回字段

同步排查重点：

- `GET /api/mini/items?side=supply&includeDeleted=1` 或 `side=demand` 是否返回目标记录。
- 返回 JSON 顶层是否是 `{ items: [...] }`。
- `POST/PUT` 是否返回 `{ item: ... }`。
- 请求头 `x-juzhen-token` 是否校验通过。
- `side` 是否为 `supply` 或 `demand`。
- `scope` 是否按 `company <-> company`、`own <-> mine`、`shared <-> shared` 映射。
- `ownerInfo` 是否保存和返回为顶层字段。
- `image` / `video` 是否为 http/https URL，列表接口不要返回 base64。
- 小程序本地缓存是否因为数据过大触发 `setStorageSync fail`。

## Git 提交规则

生成 Git 提交信息时遵循：

```text
<type>(<scope>): <subject>
```

type 取值：

```text
feat / fix / refactor / style / docs / test / chore / perf
```

要求：

- scope 使用影响模块或页面名称。
- subject 使用中文，不超过 50 字。
- 不以句号结尾。
- 每个提交只包含一个逻辑变更。
- 多模块、多类型改动要拆分提交。

## 交付标准

完成任何任务时自检：

- 改动是否可运行：至少做过相关验证。
- 改动是否可理解：关键决策有注释或文档说明。
- 改动是否可回退：模块变更边界清晰，不与无关改动耦合。

最终回复要说明：

- 改了什么。
- 验证了什么。
- 有哪些未验证或残余风险。
