# 小程序推送功能接入方案

## 一、推送类型总览

| 推送场景 | 触发时机 | 推送对象 | 优先级 |
|----------|----------|----------|--------|
| **审批通知** | 有待审批事项时 | 对应管理员/超级管理员 | P0 |
| **货源/需求更新通知** | 任何货源/需求新增/状态变更时 | 所有已认证用户（含管理员） | P1 |
| **跟进提醒** | 跟进满1小时未更新状态时 | 跟进人 + 其组管理员 | P0 |
| **逾期更新提醒** | 发布后超过7天无更新（每日8:00检查） | 发布者本人 | P1 |
| **成交确认提醒** | 货源被标记为已售出/已完成时 | 发布者本人（确认是否真实成交） | P0 |
| **需求匹配提醒** | 新需求发布时，匹配到在售货源时 | 相关货源发布者 | P1 |
| **竞争截胡提醒** | 用户跟进的货源被其他人标记成交 | 原跟进人 | P1 |
| **批量操作结果** | 批量导入/批量操作完成后 | 操作人 | P1 |
| **看板长期未访问** | 连续3天未查看看板（钝化提醒，非每日） | 普通用户 | P2 |
| **周报推送** | 每周一早上9:00 | 所有用户（按角色生成不同报告） | P2 |
| **数据异常告警** | 检测到异常数据（重复/格式错误）时 | 超级管理员 | P0 |

## 二、微信订阅消息机制

### 2.1 核心原理

微信订阅消息（SubscribeMessage）的限制：
- **必须用户主动授权**：每次推送前需调用 `wx.requestSubscribeMessage` 获取用户许可
- **有效期有限**：一次性订阅仅当次有效，长期订阅需用户勾选"总是保持以上选择"
- **模板消息 vs 订阅消息**：本方案使用订阅消息，模板消息（form_id）已不推荐

### 2.2 用户授权时机

建议在以下用户操作时静默请求授权（不在操作流程中打断用户）：

1. **用户首次打开小程序** → 弹窗"开启消息通知，审批和跟进不再错过"
2. **用户点击"跟进"按钮时** → 同时请求"跟进提醒"相关模板授权
3. **用户提交审核时** → 同时请求"审批结果通知"授权

```javascript
// 统一封装授权函数
function requestMessageSubscription(tmplIds) {
  return new Promise((resolve) => {
    wx.getSetting({
      withSubscriptions: true,
      success: (res) => {
        const maintained = res.subscriptionsSetting?.mainSettings || {};
        const missing = tmplIds.filter(id => !maintained[id]);
        if (missing.length === 0) {
          resolve(true); // 已全部授权
          return;
        }
        wx.requestSubscribeMessage({
          tmplIds: missing,
          success: () => resolve(true),
          fail: () => resolve(false)
        });
      },
      fail: () => resolve(false)
    });
  });
}
```

## 三、消息模板配置（需在小程序后台操作）

登录 **微信公众平台 → 设置与服务 → 订阅消息 → 添加模板**，按以下关键词组合申请：

### 3.1 模板一：审批通知
- 模板标题：`审核结果通知`
- 关键词：审批类型、申请人、审批状态、审批时间

### 3.2 模板二：更新通知
- 模板标题：`库存动态提醒`
- 关键词：标题、内容摘要、更新时间、操作人

### 3.3 模板三：跟进提醒
- 模板标题：`跟进提醒`
- 关键词：跟进人、货源/需求标题、提醒时间、待处理事项

### 3.4 模板四：周报推送
- 模板标题：`业绩周报`
- 关键词：姓名、周期、数据摘要、查看详情链接

### 3.5 模板五：逾期/未访问/截胡提醒
- 模板标题：`信息更新提醒`
- 关键词：信息类型、标题、逾期天数、待处理事项
- 复用：也可用于看板长期未访问提醒、竞争截胡提醒

### 3.6 模板六：成交确认提醒
- 模板标题：`审核结果通知`
- 关键词：审批类型、申请人、审批状态、审批时间
- 说明：复用审批通知模板

### 3.7 模板七：批量操作结果 / 数据异常告警
- 模板标题：`审核结果通知`
- 关键词：审批类型、申请人、审批状态、审批时间
- 说明：批量导入结果、数据异常均复用此模板

申请通过后，记录每个模板的 **TemplateId**，填入后端配置文件。

## 四、消息触发场景总表（完善版）

### 4.1 发送订阅消息

**接口**：POST `/api/mini/subscribe/send`

**请求参数**：
```json
{
  "openid": "oXXXXX",
  "templateId": "TM0001",
  "page": "pages/detail/index?id=xxx&type=supply",
  "data": {
    "thing1": { "value": "审批通知" },
    "thing2": { "value": "李诚提交了货源上传申请" },
    "phrase3": { "value": "待审核" },
    "date4": { "value": "2026-05-30 21:30" }
  }
}
```

**实现要点**：
- 调用微信 `POST https://api.weixin.qq.com/cgi-bin/message/subscribe/bizsend`
- 需要 `access_token`（7200秒有效期，需缓存并自动刷新）
- `openid` 需通过 `wx.login` + `code2session` 获取，并存储在用户记录中

### 4.2 消息触发场景对应关系

| 业务事件 | 触发函数 | 通知对象 | 模板ID | 优先级 |
|----------|----------|----------|--------|--------|
| 注册审批申请 | `approveRegistrationRequest` | 对应组管理员 | TM0001 | P0 |
| 货源上传待审核 | `requestCompleteItem` | 组管理员 | TM0001 | P0 |
| 需求上传待审核 | `requestCompleteDemand` | 组管理员 | TM0001 | P0 |
| 完成审核通过 | `approveCompletionRequestToCloud` | 申请人 + 组管理员 | TM0001 | P0 |
| 下架审核通过 | `approveOfflineRequestToCloud` | 申请人 | TM0001 | P0 |
| 货源/需求被标记售出 | `updateStatus → SOLD` | 发布者本人（确认成交） | TM0001 | P0 |
| 新增/更新货源/需求 | `upsertItemToCloud` | 所有用户 | TM0002 | P1 |
| 新需求发布，匹配在售货源 | `upsertDemandToCloud` | 相关货源发布者 | TM0002 | P1 |
| 跟进的货源被他人标记成交 | `updateStatus → SOLD` | 原跟进人 | TM0002 | P1 |
| 跟进满1小时 | 定时任务（每30分钟） | 跟进人 + 组管理员 | TM0003 | P0 |
| 货源/需求逾期7天未更新 | 定时任务（每日8:00） | 发布者本人 | TM0005 | P1 |
| 批量导入/操作完成 | 批量操作完成回调 | 操作人 | TM0002 | P1 |
| 看板连续3天未访问 | 定时任务（每日10:00） | 普通用户 | TM0005 | P2 |
| 数据异常（重复/格式错误） | 数据校验层 | 超级管理员 | TM0001 | P0 |
| 周报推送 | 定时任务（每周一9:00） | 所有用户 | TM0004 | P2 |

## 五、成交确认提醒（P0）

> 场景：货源被标记为"已售出/已完成"时，推送发布者确认是否真实成交，防止误操作

### 5.1 字段依赖

现有 `inventory.js` 中 `updateStatus` 函数已支持状态变更。需新增字段：
```javascript
// 货源
soldConfirmationSent: boolean,  // 是否已发送成交确认提醒（避免重复）
soldConfirmationToken: string,  // 确认链接Token，用于验证点击
soldAt: string                   // 现有字段，标记售出时间
```

### 5.2 触发时机

在 `inventory.js` 的 `updateStatus` 中，当状态变为 `SOLD` 时：
```javascript
// 当任意状态 → SOLD 时
if (nextStatus === INVENTORY_STATUS.SOLD) {
  next.soldConfirmationSent = false;  // ← 新增
  next.soldConfirmationToken = generateToken();  // ← 新增
  // soldAt 已存在
  // 触发确认提醒（异步，不阻塞状态更新）
  scheduleSoldConfirmationReminder(item.id);
}
```

### 5.3 确认流程设计

```javascript
// 发送确认提醒（延迟5分钟发送，避免快速操作误触）
async function scheduleSoldConfirmationReminder(itemId) {
  const item = getItemById(itemId);
  if (!item || item.soldConfirmationSent) return;

  const creator = getUserById(item.creatorId || item.sellerId);
  if (!creator || !creator.openid) return;

  await sleep(5 * 60 * 1000); // 等待5分钟

  // 再次检查，防止用户在这5分钟内撤销
  const current = getItemById(itemId);
  if (current.status !== INVENTORY_STATUS.SOLD) return;

  const confirmUrl = `pages/detail/index?id=${itemId}&type=supply&confirmSold=1&token=${item.soldConfirmationToken}`;

  await sendSubscribeMessage(creator.openid, TEMPLATE_IDS.SOLD_CONFIRM, {
    thing1: { value: "货源" },
    thing2: { value: item.title.slice(0, 20) },
    phrase3: { value: "已标记售出" },
    date4: { value: formatDateTime(item.soldAt) },
    thing5: { value: "请点击确认是否真实成交" }
  }, confirmUrl);

  updateItemConfirmationSent(itemId);
}
```

用户点击消息后，进入详情页弹窗确认：
- **确认成交** → 状态保持 SOLD，结束
- **误操作** → 状态回退到上一状态（需记录 `previousStatus`），清除 soldAt

### 5.4 防重复发送

使用 `soldConfirmationSent` 字段标记，已发送则不再发。

## 六、需求匹配提醒（P1）

> 场景：新需求发布时，根据类别、品牌、型号等维度匹配在售货源，推送给相关发布者

### 6.1 匹配规则

```javascript
function findMatchingSuppliers(demand) {
  const { category, brand, model, businessType, rentalBudgetMin, rentalBudgetMax } = demand;

  return getItems().filter(item => {
    if (item.isDeleted || item.status === INVENTORY_STATUS.SOLD) return false;

    // 业务类型匹配（求购 vs 在售）
    if (!isBusinessMatch(item.businessType, demand.businessType)) return false;

    // 品类匹配（宽松：品牌或型号包含即可）
    const brandMatch = !brand || item.brand?.includes(brand) || brand.includes(item.brand);
    const modelMatch = !model || item.model?.includes(model) || model.includes(item.model);
    if (!brandMatch && !modelMatch) return false;

    // 租金预算匹配（如果需求有预算上限）
    if (isRentDemand(demand) && isRentable(item)) {
      if (!matchRentalBudget(item, demand)) return false;
    }

    // 排除自己发布的需求（不要推给自己）
    if (item.creatorId === demand.creatorId) return false;

    return true;
  });
}
```

### 6.2 发送逻辑

```javascript
// 在 demand.js 的 upsertDemandToCloud 成功后调用
async function notifyMatchingSuppliers(demand) {
  const matchedItems = findMatchingSuppliers(demand);
  if (matchedItems.length === 0) return;

  const demandCreator = getUserById(demand.creatorId);

  for (const item of matchedItems) {
    const owner = getUserById(item.creatorId || item.sellerId);
    if (!owner || !owner.openid) continue;

    await sendSubscribeMessage(owner.openid, TEMPLATE_IDS.MATCH_NOTIFY, {
      thing1: { value: "需求匹配" },
      thing2: { value: demand.title.slice(0, 20) },
      thing3: { value: item.title.slice(0, 20) },
      phrase4: { value: "有新增需求可跟进" }
    }, `pages/detail/index?id=${demand.id}&type=demand`);
  }
}
```

### 6.3 节流控制

同一需求匹配同一货源发布者，最多推送1次/天（按 `demandId + ownerId` 去重）。

## 七、竞争截胡提醒（P1）

> 场景：业务员A跟进了某货源，业务员B抢先标记为成交。此时推送A："您跟进的XXX已被其他人成交，可选择其他货源跟进"

### 7.1 字段依赖

现有字段已足够：
- `followOwnerId` - 原跟进人
- `followOwnerName` - 原跟进人姓名
- `sellerId` / `creatorId` - 货源发布者

### 7.2 触发时机

在 `inventory.js` 的 `updateStatus` 中，当状态变为 `SOLD` 时：
```javascript
if (nextStatus === INVENTORY_STATUS.SOLD) {
  const currentItem = getItemById(itemId);

  // 如果有跟进人且跟进人不是当前操作人
  if (currentItem.followOwnerId &&
      currentItem.followOwnerId !== user.id &&
      currentItem.status === INVENTORY_STATUS.FOLLOWING) {

    // 延迟10分钟发送（等成交确认完成后）
    scheduleCompetitiveNotification(itemId, currentItem.followOwnerId);
  }
}
```

### 7.3 发送逻辑

```javascript
async function scheduleCompetitiveNotification(itemId, followOwnerId) {
  await sleep(10 * 60 * 1000);

  const item = getItemById(itemId);
  if (item.status !== INVENTORY_STATUS.SOLD) return; // 如果撤销了就不发

  const followOwner = getUserById(followOwnerId);
  if (!followOwner || !followOwner.openid) return;

  const soldBy = getUserById(item.sellerId || item.creatorId);
  const soldByName = soldBy ? soldBy.name : "其他人";

  await sendSubscribeMessage(followOwner.openid, TEMPLATE_IDS.COMPETITIVE_ALERT, {
    thing1: { value: "截胡提醒" },
    thing2: { value: item.title.slice(0, 20) },
    phrase3: { value: "已被" + soldByName + "成交" },
    thing4: { value: "可查看其他待跟进货源" }
  }, "pages/board/index?status=following");
}
```

## 八、批量操作结果通知（P1）

> 场景：批量导入货源/需求完成后，推送导入结果（成功数、失败数、失败原因）

### 8.1 触发时机

在 `batch-import/index.js` 的导入流程完成后：
```javascript
async function handleBatchImportResult(filePath, operatorId) {
  const result = await processBatchImport(filePath);
  const operator = getUserById(operatorId);

  if (!operator || !operator.openid) return;

  const successCount = result.successCount || 0;
  const failCount = result.failCount || 0;
  const failReasons = (result.failItems || []).slice(0, 3); // 只显示前3条

  let content = `成功${successCount}条`;
  if (failCount > 0) {
    content += `，失败${failCount}条`;
  }

  await sendSubscribeMessage(operator.openid, TEMPLATE_IDS.BATCH_RESULT, {
    thing1: { value: "批量导入结果" },
    thing2: { value: content },
    phrase3: { value: failCount > 0 ? "有失败记录，请查看" : "全部成功" },
    thing4: { value: formatDateTime(new Date()) }
  }, `pages/batch-import/index?resultId=${result.resultId}`);
}
```

### 8.2 消息内容示例

- 成功5条，失败2条：`成功5条，失败2条，有失败记录，请查看`

## 九、看板长期未访问提醒（P2）

> 场景：普通用户连续3天未打开看板，发送一次钝化提醒。不是每日重复，避免骚扰

### 9.1 字段依赖

用户表新增：
```javascript
lastBoardVisitAt: string,  // 最近查看看板的时间（ISO）
staleReminderSent: boolean  // 是否已发送长期未访问提醒
```

### 9.2 追踪访问时间

在 `pages/board/index.js` 的 `onShow` 中更新：
```javascript
onShow() {
  updateUserLastBoardVisit(app.globalData.activeUserId);
  // ...其他逻辑
}

function updateUserLastBoardVisit(userId) {
  updateUser(userId, {
    lastBoardVisitAt: new Date().toISOString(),
    staleReminderSent: false  // 重新访问后重置，允许下次再提醒
  });
}
```

### 9.3 定时检查逻辑

```javascript
// 云函数：checkInactiveUsers（每日10:00触发）
async function checkInactiveUsers() {
  const THREE_DAYS = 3 * 24 * 60 * 60 * 1000;
  const threeDaysAgo = new Date(Date.now() - THREE_DAYS).toISOString();

  const inactiveUsers = getUsers().filter(user => {
    if (!user.lastBoardVisitAt) return true; // 从未访问过也算
    return user.lastBoardVisitAt < threeDaysAgo && !user.staleReminderSent;
  });

  for (const user of inactiveUsers) {
    if (!user.openid) continue;

    await sendSubscribeMessage(user.openid, TEMPLATE_IDS.INACTIVE_REMINDER, {
      thing1: { value: "看板提醒" },
      thing2: { value: "您已3天未查看看板" },
      phrase3: { value: "有新的商机等待跟进" },
      thing4: { value: "点击查看最新货源" }
    }, "pages/board/index");

    updateUser(user.id, { staleReminderSent: true });
  }
}
```

### 9.4 定时触发配置

```json
{
  "name": "inactive-user-trigger",
  "trigger": {
    "type": "timer",
    "config": "0 0 10 * * *"  // 每天早上10:00
```

## 十、数据异常告警（P0）

> 场景：超级管理员需要及时知道数据异常，如重复货源、格式错误、必填字段缺失等

### 10.1 异常检测规则

```javascript
const ANOMALY_RULES = [
  {
    id: "DUPLICATE_TITLE",
    check: (item) => {
      const sameTitle = getItems().filter(i =>
        i.id !== item.id && !i.isDeleted && i.title === item.title
      );
      return sameTitle.length > 0;
    },
    message: "存在相同标题的货源"
  },
  {
    id: "MISSING_PRICE",
    check: (item) => {
      return !item.price && !item.rentalPrices?.oneYearFull;
    },
    message: "价格和租金均为空"
  },
  {
    id: "INVALID_CONTACT",
    check: (item) => {
      return !item.contactName || !item.contactMethod;
    },
    message: "联系方式缺失"
  },
  {
    id: "FUTURES_WITHOUT_ARRIVAL",
    check: (item) => {
      return item.sourceType === SOURCE_TYPES.FUTURES && !item.arrivalDate;
    },
    message: "期货货源缺少预计到货时间"
  }
];
```

### 10.2 告警发送逻辑

```javascript
// 在 upsertItemToCloud / upsertDemandToCloud 中，数据落地前检测
async function checkAndAlertAnomalies(item, side) {
  const anomalies = [];

  for (const rule of ANOMALY_RULES) {
    if (rule.check(item)) {
      anomalies.push({
        ruleId: rule.id,
        message: rule.message,
        itemId: item.id,
        itemTitle: item.title,
        side
      });
    }
  }

  if (anomalies.length === 0) return;

  const superadmins = getSuperadmins();
  for (const admin of superadmins) {
    if (!admin.openid) continue;

    const msg = anomalies.length === 1
      ? anomalies[0].message
      : `检测到${anomalies.length}项数据异常`;

    await sendSubscribeMessage(admin.openid, TEMPLATE_IDS.ANOMALY_ALERT, {
      thing1: { value: "数据异常告警" },
      thing2: { value: msg },
      phrase3: { value: "请及时处理" },
      date4: { value: formatDateTime(new Date()) }
    }, `pages/admin-approvals/index?anomaly=1`);
  }
}
```

## 十一、逾期更新提醒实现方案

> 场景：用户上传的货源/需求信息超过7天不更新，订阅消息推送提醒更新看板信息

### 11.1 字段依赖

现有 `inventory.js` 和 `demand.js` 已有 `updatedAt` 字段，每次更新会自动写入，无需新增字段。

### 11.2 逾期检查逻辑（定时任务）

```javascript
// 云函数：checkStaleItems（每日8:00触发一次）
async function checkStaleItems() {
  const now = Date.now();
  const SEVEN_DAYS = 7 * 24 * 60 * 60 * 1000;
  const yesterday = new Date(now - SEVEN_DAYS).toISOString();

  // 找出逾期7天未更新的有效记录（已上线/跟进中）
  const staleItems = getItems().filter(item => {
    if (item.isDeleted || item.status === INVENTORY_STATUS.SOLD) return false;
    const lastUpdate = item.updatedAt || item.createdAt;
    return lastUpdate && lastUpdate < yesterday;
  });

  const staleDemands = getDemands().filter(item => {
    if (item.isDeleted) return false;
    const lastUpdate = item.updatedAt || item.createdAt;
    return lastUpdate && lastUpdate < yesterday;
  });

  // 按发布者分组，避免重复发送
  const userNotifications = new Map();

  for (const item of staleItems) {
    const creatorId = item.creatorId || item.sellerId;
    if (!creatorId) continue;
    if (!userNotifications.has(creatorId)) {
      userNotifications.set(creatorId, []);
    }
    userNotifications.get(creatorId).push({
      type: "supply",
      id: item.id,
      title: item.title,
      updatedAt: item.updatedAt
    });
  }

  for (const item of staleDemands) {
    const creatorId = item.creatorId;
    if (!creatorId) continue;
    if (!userNotifications.has(creatorId)) {
      userNotifications.set(creatorId, []);
    }
    userNotifications.get(creatorId).push({
      type: "demand",
      id: item.id,
      title: item.title,
      updatedAt: item.updatedAt
    });
  }

  // 发送订阅消息
  for (const [userId, items] of userNotifications) {
    const user = getUserById(userId);
    if (!user || !user.openid) continue;

    const count = items.length;
    const latestItem = items.reduce((a, b) =>
      (a.updatedAt || "") < (b.updatedAt || "") ? a : b
    );

    await sendSubscribeMessage(user.openid, TEMPLATE_IDS.STALE_REMINDER, {
      thing1: { value: "货源/需求" },
      thing2: { value: latestItem.title.slice(0, 20) + (latestItem.title.length > 20 ? "..." : "") },
      number3: { value: String(count) },
      thing4: { value: "点击查看全部逾期记录" },
      date5: { value: formatDate(latestItem.updatedAt) }
    }, `pages/board/index?userId=${userId}`);
  }
}
```

## 十二、跟进提醒实现方案（1小时自动推送）

> 场景：业务员跟进的货源/需求超过1小时未更新状态，提醒跟进人及组管理员

### 12.1 字段准备

现有字段（已在 inventory.js 中）：
- `followOwnerId` - 跟进人ID
- `followOwnerName` - 跟进人姓名
- `lastFollowedAt` - 标记跟进的时间点
- `followReminderSent` - 是否已发送提醒标记
- `followOwnerGroup` - 跟进人所在组

### 12.2 标记跟进时写入时间戳

```javascript
// 在 inventory.js 的 updateStatus 函数中
// 当 status → FOLLOWING 时
if (nextStatus === INVENTORY_STATUS.FOLLOWING) {
  next.followOwnerId = user.id;
  next.followOwnerName = user.name;
  next.lastFollowedAt = new Date().toISOString();  // ← 新增
```

### 12.3 定时检查逻辑（每30分钟）

```javascript
// 云函数：checkFollowReminders（每30分钟触发一次）
async function checkFollowReminders() {
  const now = Date.now();
  const ONE_HOUR = 60 * 60 * 1000;

  // 找出超时的跟进记录
  const overdueItems = getItems().filter(item => {
    if (!item.lastFollowedAt || item.followReminderSent) return false;
    const elapsed = now - new Date(item.lastFollowedAt).getTime();
    return elapsed > ONE_HOUR && item.status === INVENTORY_STATUS.FOLLOWING;
  });

  const overdueDemands = getDemands().filter(item => {
    if (!item.lastFollowedAt || item.followReminderSent) return false;
    const elapsed = now - new Date(item.lastFollowedAt).getTime();
    return elapsed > ONE_HOUR && item.status === "following";
  });

  for (const item of [...overdueItems, ...overdueDemands]) {
    const user = getUserById(item.followOwnerId);
    if (!user) continue;

    // 发送消息给跟进人
    if (user.openid) {
      await sendSubscribeMessage(user.openid, TEMPLATE_IDS.FOLLOW_REMINDER, {
        thing1: { value: user.name },
        thing2: { value: item.title },
        date3: { value: formatDateTime(new Date()) },
        thing4: { value: "已跟进超过1小时，请及时更新状态" }
      });
    }

    // 发送消息给组管理员
    const admin = getGroupAdmin(user.group);
    if (admin && admin.openid) {
      await sendSubscribeMessage(admin.openid, TEMPLATE_IDS.FOLLOW_REMINDER, {
        thing1: { value: user.name },
        thing2: { value: item.title },
        date3: { value: formatDateTime(new Date()) },
        thing4: { value: "已跟进超过1小时，请督促更新状态" }
      });
    }

    // 标记已发送
    if (item.side === "supply") {
      updateItemReminderSent(item.id);
    } else {
      updateDemandReminderSent(item.id);
    }
  }
}
```

## 十三、周报自动生成方案

### 13.1 数据统计口径

**个人周报**（每周一推送）：
- 本周新增发布数（supply + demand）
- 本周成交数（status=sold/done）
- 正在跟进数（status=following）
- 成交率 = 成交数 / 总发布数
- 相比上周变化（增减百分点）

**管理员/超级管理员周报**：
- 包含上述个人数据
- 团队汇总：本周团队总发布、成交、跟进
- 待审批数：注册待审批 + 货源/需求待审批
- 新增注册人数
- 逾期未更新记录（跟进超过7天无操作）

### 13.2 周报生成函数

```javascript
async function generateWeeklyReports() {
  const users = getApprovedUsers();
  const weekStart = getWeekStartDate(); // 本周一 00:00
  const weekEnd = new Date();           // 当前时间

  for (const user of users) {
    const report = buildWeeklyReport(user, weekStart, weekEnd);

    if (user.role === "staff") {
      const personal = buildPersonalReport(user, weekStart, weekEnd, report);
      if (user.openid) {
        await sendSubscribeMessage(user.openid, TEMPLATE_IDS.WEEKLY_REPORT, {
          thing1: { value: user.name },
          date2: { value: `${formatDate(weekStart)}~${formatDate(weekEnd)}` },
          thing3: { value: `发布${personal.publish} | 成交${personal.sold} | 跟进${personal.following}` },
          thing4: { value: personal.rateChange >= 0 ? `成交率↑${personal.rateChange}%` : `成交率↓${Math.abs(personal.rateChange)}%` }
        });
      }
    } else if (user.role === "admin" || user.role === "superadmin") {
      const business = buildBusinessReport(user, weekStart, weekEnd, report);
      if (user.openid) {
        await sendSubscribeMessage(user.openid, TEMPLATE_IDS.WEEKLY_REPORT, {
          thing1: { value: user.name },
          date2: { value: `${formatDate(weekStart)}~${formatDate(weekEnd)}` },
          thing3: { value: `团队发布${business.teamPublish} | 团队成交${business.teamSold} | 待审${business.pendingCount}` },
          thing4: { value: `新增注册${business.newUsers}人` }
        });
      }
    }
  }
}
```

## 十四、access_token 管理

微信 access_token 是调用所有微信 API 的凭证，有效期7200秒，必须：

1. **集中管理**：在服务端（或云函数）统一获取和刷新
2. **缓存使用**：缓存至 Redis 或全局变量，快过期时自动刷新
3. **绝不泄露 appsecret**：appsecret 只存在于后端，绝不传送到小程序前端

```javascript
// 云函数：getAccessToken
let cachedToken = null;
let tokenExpireAt = 0;

async function getAccessToken() {
  if (cachedToken && Date.now() < tokenExpireAt - 60000) {
    return cachedToken; // 提前1分钟刷新
  }
  const res = await wx.cloud.callContainer({
    config: { env: process.env.CLOUD_ENV },
    path: '/api/token',
    method: 'GET'
  });
  cachedToken = res.access_token;
  tokenExpireAt = Date.now() + (res.expires_in || 7200) * 1000;
  return cachedToken;
}
```
