## 2. 产品概述

### 2.1 产品名称
货源看板小程序

### 2.2 产品定位
货源看板小程序是一款面向内部员工使用的轻量化业务协同工具，主要用于统一管理货源信息的录入、查看、跟进、状态更新和操作留痕。产品依托微信小程序形态，降低使用门槛，提升业务协同效率，并为后续统计分析、大屏展示和系统对接打下基础。

### 2.3 建设背景
当前货源信息若通过腾讯文档、Excel 或群消息方式管理，虽然上线快、成本低，但在实际业务使用中会逐步暴露以下问题：

1. 字段填写口径不统一，容易出现漏填、错填、描述不规范的问题。
2. 仅能记录数据，无法承载注册、审批、状态流转等业务流程。
3. 权限控制较粗，缺少基于角色的差异化管理能力。
4. 信息更新过程缺少留痕，后续复盘和责任追踪困难。
5. 数据可读性尚可，但结构化程度有限，不利于后续统计、展示和系统对接。

因此，有必要建设一个轻量、低门槛、可扩展的小程序系统，将货源管理从“文档记录”升级为“流程化、结构化、可持续扩展的业务工具”。

### 2.4 产品目标
本期 MVP 的目标如下：

1. 建立内部员工统一使用的货源信息入口。
2. 搭建员工注册申请与管理员审批流程。
3. 支持货源信息新增、编辑、查看、状态更新和逻辑删除。
4. 支持看板式展示、搜索筛选和个人工作台能力。
5. 建立日志留痕机制，为业务管理和后续复盘提供依据。
6. 预留大屏展示接口，为未来展示场景和系统化升级做好准备。

## 3. 产品价值

### 3.1 业务价值
1. 提升货源信息录入、共享和跟进效率。
2. 降低因表格式维护造成的信息口径不一致问题。
3. 让货源状态透明化，方便团队协同推进。
4. 让业务数据逐步沉淀，形成后续可分析、可展示、可扩展的基础能力。

### 3.2 相比腾讯文档等方案的优势
1. 小程序不是通用表格，而是按业务流程定制，能承载注册、审批、录入、跟进、成交和留痕的完整闭环。
2. 字段和录入规则统一，可从源头减少漏填、乱填、口径不一致的问题。
3. 权限控制更细，不只是编辑和只读，还可以按员工、管理员、角色分层开放能力。
4. 数据天然结构化，后续做搜索、统计、日志追踪、大屏展示和系统打通更方便。
5. 使用体验更贴近业务场景，员工在手机端即可直接完成操作，不需要人工维护表格结构。
6. 未来扩展性更强，可逐步延伸到消息通知、企业微信打通、后台系统、数据库、大屏端等能力，不需要推翻重来。

## 4. 目标用户与角色

### 4.1 员工
内部业务人员，负责注册申请、查看货源、录入货源、更新状态、查看个人相关数据。

### 4.2 管理员
系统管理人员，负责审批员工注册申请，并承担演示和管理入口职责。

### 4.3 管理层
作为数据查看方，可通过看板或后续大屏能力了解整体货源盘面和业务动态。

---

## 5. 使用场景

1. 新员工首次进入小程序，提交注册申请，等待管理员审核。
2. 审批通过后，员工可进入货源看板查看当前货源信息。
3. 员工录入新货源，填写设备型号、配置、数量、价格、来源和交付日期等信息。
4. 业务人员将某条货源标记为“跟进中”，明确当前跟进责任。
5. 成交后将货源更新为“已售出”，系统自动记录成交人和成交时间。
6. 暂时无法出售的货源，可更新为“已下架”。
7. 管理人员通过日志查看货源变更历史，便于复盘和管理。
8. 后续可将货源数据同步到展示大屏，用于会议室或业务现场展示。

## 7. 详细功能说明

## 7.1 注册申请模块

### 功能目标
为首次使用系统的内部员工提供身份登记入口，确保系统仅对授权人员开放。

### 功能描述
1. 用户首次进入系统后默认进入注册页面。
2. 用户填写姓名、手机号、部门、职位、备注等基础信息。
3. 提交后生成一条待审批申请记录。
4. 系统记录最近一次提交注册的手机号，便于再次进入时快速识别申请状态。
5. 如手机号已存在正式账号，系统提示不可重复申请。
6. 如手机号已有待审批申请，则不重复创建新的申请单。

### 输入字段
1. 姓名
2. 手机号
3. 部门
4. 职位
5. 备注

### 状态说明
1. `none`：未提交
2. `pending`：待审批
3. `approved`：已通过
4. `rejected`：已拒绝

### 当前限制
当前版本中，管理员微信通知仍为预留状态，尚未接入正式消息能力。

---

## 7.2 管理员审批模块

### 功能目标
用于管理员审核员工注册申请，决定是否开通系统使用权限。

### 功能描述
1. 管理员可查看全部待审批申请。
2. 每条申请展示申请人姓名、手机号、部门、职位、申请时间等信息。
3. 管理员可对申请执行“通过”或“拒绝”操作。
4. 审批通过后，系统自动生成对应员工账号。
5. 审批拒绝后，申请记录更新为已拒绝状态。
6. 非管理员角色不可访问该页面。

### 输出结果
1. 通过：生成正式用户账号，申请状态变更为已通过。
2. 拒绝：申请状态变更为已拒绝。

## 7.3 货源看板模块

### 功能目标
提供货源信息总览，便于员工快速掌握当前盘面并进行检索。

   1. 在售数量
   2. 跟进中数量

4. 支持按状态筛选。
5. 支持下拉刷新。
6. 支持点击单条记录进入详情页。

3. 所在地
4. 配置摘要
5. CPU
6. 内存
7. 存储
8. GPU
9. 网卡

### 状态筛选项
1. 全部
2. 在售
3. 跟进中
4. 已售出
5. 已下架

## 7.4 货源详情模块

### 功能目标
展示单条货源的完整信息，并支持状态管理、编辑和逻辑删除操作。

### 功能描述
1. 查看货源完整字段信息。
2. 查看该货源的操作日志列表。
3. 支持进入编辑页修改货源内容。
4. 支持将货源标记为“跟进中”。
5. 支持将货源标记为“已售出”。
6. 支持将货源标记为“已下架”。
7. 支持逻辑删除。

### 状态变更规则
1. 标记为“跟进中”时，系统记录当前操作人为跟进负责人。
2. 标记为“已售出”时，系统记录成交人、成交时间，并同步保留跟进人信息。
3. 标记为“已下架”时，系统记录状态变更日志。
4. 删除后货源不在看板中展示，但历史日志继续保留。

## 11. 核心业务流程

### 11.1 注册与审批流程
1. 员工进入小程序。
2. 填写注册信息并提交。
3. 系统生成待审批申请。
4. 管理员进入审批页进行审核。
5. 审批通过后生成正式账号。
6. 员工可重新进入系统访问货源看板。

### 11.2 货源录入流程
1. 员工进入新增货源页面。
2. 填写货源基本信息与配置字段。
3. 点击保存。
4. 系统生成货源记录。
5. 系统自动生成新增日志。
6. 新货源显示在看板列表中。

### 11.3 货源跟进流程
1. 员工进入货源详情页。
2. 点击“标记跟进中”。
3. 系统更新状态。
4. 系统记录当前跟进人。
5. 系统生成状态变更日志。

### 11.4 成交流程
1. 员工进入货源详情页。
2. 点击“标记已售出”。
3. 系统更新货源状态。
4. 系统记录成交人和成交时间。
5. 系统生成成交日志。

### 11.5 删除流程
1. 员工进入货源详情页。
2. 发起删除操作。
3. 系统弹出确认提示。
4. 确认后执行逻辑删除。
5. 该货源不再在看板中展示。
6. 历史日志继续保留。
